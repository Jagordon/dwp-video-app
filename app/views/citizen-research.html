Joining an appointment | Browser check | Waiting message

Testing link 1 - June 2021
Assumptions
  1. Users will want to know as soon as possible (before the start page) if they are on an unsupported browser.
  2. If accessing from UC, showing the start page before/at the same time as the browser error will help the user to understand it is the video service that is unsupported.
  3. We should not completely stop the users journey if we don’t know that they will have a bad experience.
  4. For when the browser is Chime supported but not tested internally, or an outdated version, a banner on the start page will be sufficient to warn the user.
  5. If the browser may be unsupported, users may have ‘banner blindness’ and therefore need a specific page.
  6. For browsers we know are unsupported the user should not be allowed to continue to the video service.
  7. Providing a ‘copy link’ button will allow the user to quickly switch to a supported browser, rather than having to go back into UC on the new browser.
Hypotheses

Browser check

Because: we know some attendees try to join video appointments on unsupported browsers
We believe that: warning users they may have issues on their current browser, before they enter the appointment
Will result in: a better user experience and more successful appointments
We'll know this is true: when we see users are attending on supported browsers, and a reduction in failed appointments.

Waiting message

Because:

  • we know when some users join the meeting first, they use the refresh button multiple times before the other person joins
  • we know some users do not like seeing the large view of themselves as it makes them uncomfortable.
We believe that:
  • by explaining they have joined the appointment, but the other person hasn't
  • by making the large view of themselves a smaller 'selfie' view
Will result in:
  • users understanding what is happening and that they do not need to do anything
  • users feeling more comfortable
We'll know this is true:
  • when users give us feedback that they understand what is happening
  • when users give us feedback that they feel more comfortable with a smaller 'selfie' view
  • if there is a reduction in the use of the refresh button prior to the second person joining.

Multiple attendees | Replace background

Combined testing link 1 - March 2021
Hypotheses

Small selfie

Because: we know not all users feel comfortable seeing themselves on screen and need a clear view of other participants
We believe that: showing a small view of themselves in the corner of the screen
Will result in: users feeling at ease and able to focus on the other participants in the appointment
We'll know this is true: when users state this in user testing.

Equal selfie

Because: users need to ensure they can be seen clearly
We believe that: showing all feeds, including themselves, in a grid format
Will result in: users being seen clearly throughout the appointment
We'll know this is true: when users choose this option in user testing over the alternatives.

Fill frame

Because: we know people value seeing facial expressions
We believe that: showing a cropped, zoomed in view of other participants
Will result in: attendees being able to read facial expressions and an efficient use of screen space
We'll know this is true: when users choose this option in user testing over the alternatives.

Fit-to-frame

Because: we know users sometimes need to see the whole frame
We believe that: keeping the whole frame of other participants in view
Will result in: users being able to see everything they need to see
We'll know this is true: when users choose this option in research over the alternatives.

Blur background

Because: there is a need for users to hide their background
We believe that: blurring the background
Will result in: sufficient privacy for the user
We'll know this is true: when users state this in user testing.

Cover background

Because: there is a need for users to hide their background
We believe that: fully replacing the background with a pre-selected image
Will result in: greater privacy for the user
We'll know this is true: when users choose this option in research.

Radio buttons

Because: there is a need for users to hide their background and we assume this will be at the beginning of the appointment
We believe that: providing radio buttons once in the appointment
Will result in: understanding whether blur is on or off, and how to change it
We'll know this is true: when participants show their understanding in usability testing.

Settings drop down menu

Because: we know there is a need for a citizen to hide their background and we assume the action will only be once per appointment, if at all
We believe that: providing a settings menu
Will result in: a simple interface, with less distractions from the appointment
We'll know this is true: when participants show their understanding in usability testing.
